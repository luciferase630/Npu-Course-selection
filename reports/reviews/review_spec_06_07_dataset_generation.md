## 二、`spec/06` 生成规范审阅详情

### 2.1 已确认合理的部分

| 条目 | 评价 |
|---|---|
| **规模 40×200** | 合理。比 smoke（6×8）大 enough 观察策略差异，又比真实规模小 enough 便于调试。 |
| **7 个课程类别** | 层次分明，覆盖基础/核心/选修/通识/英语/体育/实验，贴近真实培养方案。 |
| **学分 0.5 倍数** | 约束明确，生成器容易实现。 |
| **容量带随机扰动** | 避免"同一类课容量完全相同"的不自然现象。 |
| **latent factor 生成** | teacher_quality + course_quality + category_affinity + time_affinity + profile_relevance + student_noise 六因子结构清晰，且 CSV 只输出最终 `utility`，与 MVP"不拆解效用来源"的原则一致。 |
| **必修课不硬塞进 utility** | 反复强调"必修重要性通过 requirements 和运行时惩罚表达"，与 `docs/00` 和 `context.py` 的派生惩罚模型口径一致。 |
| **eligible 范围 80–140** | 合理。200 门课中约 40–70% eligible，既保证选择空间，又避免"全 eligible"导致缺乏结构约束。 |
| **deterministic 要求** | 同一 seed 生成完全相同，便于复现和回归测试。 |

### 2.2 建议补充或明确的部分

| # | 问题 | 建议 |
|---|---|---|
| **1** | `required_profile` 格式未精确定义 | 建议明确为字符串枚举：`CS_2026`、`SE_2026`、`AI_2026`。如果生成器内部使用，需要在 `generation_metadata.json` 中记录 profile 到 requirements 的映射规则。 |
| **2** | `grade_stage` 混合比例未给出 | 若选择混合生成（而非固定 `junior`），建议给出默认比例，例如 `sophomore:junior:senior:graduation_term = 2:4:3:1`，避免生成器随意发挥导致分布不均。 |
| **3** | `profile_relevance` 的放大倍数未约束 | latent factor 公式中 `profile_relevance` 会让本专业课程 utility 偏高。建议给出上限约束（例如 `profile_relevance <= 15`），防止本专业课程 utility 系统性碾压通识课，导致实验中学生只选本专业课、通识课完全无人问津。 |
| **4** | 容量下限与竞争烈度的关系 | 大课容量 60–140，对 40 个学生来说可能意味着很多课竞争不足（`m < n`）。这不是 bug，但可能导致公式 `f(m,n)` 在大量课程上无定义（`m-n < 0`）。建议生成器在 capacity 抽样时，热门基础课/英语课适当压低至 30–60，确保足够竞争。 |
| **5** | `5-6` 午饭时段权重 0.15 | 实现带权抽样时，建议用累积分布函数（CDF）而非简单拒绝采样，避免在 200 门课规模下产生偏差或效率问题。 |

## 三、`spec/07` 分布审阅规范审阅详情

### 3.1 已确认合理的部分

| 条目 | 评价 |
|---|---|
| **硬约束清单** | 学分合法性、`5-6` 占比、`time_slot` 格式、eligible 数量范围、required 可满足性——全部可量化、可自动检查。 |
| **教师一致性检查** | "同一教师下不同学生 utility 不应极端两极化"是数据质量的关键防线。建议计算教师内 utility 的标准差或四分位距。 |
| **时间冲突压力检查** | 承认"无法严格验证组合可行性时可明确记录"，不强行要求 NP-hard 的精确解，态度务实。 |
| **deterministic 检查** | 用 hash 比较两次同 seed 输出，方法简单可靠。 |
| **reviewer 输出格式** | `通过 / 不通过 / 有条件通过` 三级结论 + 结构化章节，对下一个 agent 足够友好。 |

### 3.2 建议补充或量化的部分

| # | 问题 | 建议 |
|---|---|---|
| **1** | "教师均值不能几乎一样"缺少量化阈值 | 建议补充：计算所有 `teacher_id` 平均 utility 的变异系数（CV = std/mean）。若 CV < 0.15，视为"教师差异不足"，标为有条件通过。 |
| **2** | "utility 不应大量堆在 1 或 100"缺少量化阈值 | 建议补充：边界值（`<=5` 或 `>=95`）的边占比应 `< 5%`。若超过，视为截断过度。 |
| **3** | "11-12 不能接近空白"缺少量化阈值 | 建议补充：`11-12` 时段课程数应 `>= 5`（200 门课的 2.5%），或至少占总课次的 `>= 2%`。 |
| **4** | 时间冲突检查可以简化 | 对 40 个学生×约 5–8 个 required course_code，可以做一个轻量级检查：对每个学生的 required 集合，枚举所有教学班的时段组合，检查是否存在至少一组无冲突组合。若不存在，直接标记该学生 required 不可满足。这个检查在 40×8 规模下计算量很小。 |
| **5** | 缺少 `category` 分布均衡性检查 | `spec/06` 要求"同一 category 不应系统性挤在某一天或某个时间块"，`spec/07` 第 4 节提到了，但检查项列表中没有明确量化指标。建议补充：对每个 category，计算其课程在 30 个 `weekday-time_block` 上的分布，若任一 category 的某槽位占比超过该 category 总课次的 15%，标记为集中偏差。 |

## 四、两个 Spec 之间的交叉一致性

| 检查项 | `spec/06` 定义 | `spec/07` 验收 | 一致性 |
|---|---|---|---|
| 学分范围 | `[0.5, 7.0]`，0.5 倍数 | 硬约束：`credit*2` 为整数 | ✅ 一致 |
| eligible 数量 | 约 80–140 | 硬约束：80–140 | ✅ 一致（06 的"约"在 07 中收紧为硬约束） |
| `5-6` 占比 | `<=6%` | 硬约束：`<=6%`，超则不通过 | ✅ 一致 |
| 时间块权重 | `5-6` 权重 0.15 | 验收：占比 `<=6%` | ✅ 逻辑自洽（0.15/5.9≈2.5%，但多时段课程会稀释，实际占比应远低于 6%） |
| required 可满足性 | 对应 course_code 至少有一个 eligible 教学班 | 同 | ✅ 一致 |
| utility 范围 | `1–100` | 检查 `1–100`，不堆边界 | ✅ 一致 |
| 教师一致性 | 好老师总体高，差老师总体低 | 检查教师均值差异和内部标准差 | ✅ 呼应 |

## 五、给生成器实现 Agent 的建议

1. **先实现一个 `validate_medium_dataset()` 函数**：在生成器脚本中内嵌 `spec/07` 的核心硬约束检查（学分合法性、time_slot 格式、eligible 范围、required 可满足性），生成后立即自验，不通过则报错。这比生成后再让另一个 agent review 更高效。
2. **`required_profile` 的处理**：由于 `students.csv` 的 loader 不会读取 `required_profile`，建议生成器把 profile 到 requirements 的映射规则写进 `generation_metadata.json`，而不是依赖 CSV 中的扩展字段被下游读取。
3. **容量微调建议**：热门基础课/英语课容量建议压到 30–60（而非 60–140），确保 40 个学生中至少有 1.5–2 倍拥挤比，否则实验中的竞争信号（`m/n`）会过于稀疏。
4. **latent factor 实现顺序**：建议先生成全局的 `teacher_quality` 和 `course_quality`（与具体学生无关），再加 `category_affinity`、`time_affinity`、`profile_relevance`（学生相关但有结构），最后加 `student_noise`（纯随机）。这样便于分层调试。
