# Python + 大模型调用实验方案

本文定义后续如何用 Python 和大模型接口模拟学生投豆决策。当前阶段只写实验方案，不写代码实现。

核心目标：模拟学生在预算、轮次、容量和培养方案约束下，如何最大化自己的期望效用，而不是最大化课程数量。

## 1. 实验目标

第一阶段实验目标是观察不同学生策略在选课豆子机制下的效用结果：

- 个体是否能获得高效用课表。
- 必修课未完成惩罚是否被有效规避。
- 预算在三轮之间如何被消耗。
- 风险偏好如何影响投豆集中度。
- 学生如何在轮内多个时间点根据待选人数修改投豆。
- 同一课程代码下，不同老师和时间的课程班如何改变学生投豆。
- 大模型扮演学生时，会不会产生类似人类的保守、抢热门、赌冷门、平均撒豆等行为。

## 2. 最小合成数据集

先构造一个不依赖真实教务数据的小型数据集：

- 学生数：30到200人。
- 课程班数：10到40个。
- 每个学生预算：100豆。
- 豆子类型：只能为整数；预算、投豆、支付、退豆和剩余预算均使用整数。
- 课程类型：必修课、专业选修课、通识选修课、英语类。
- 课程容量：从小班到大课混合设置。
- 课程代码与课程班：同一 `course_code` 可以对应多个 `course_id`。
- 授课教师：每个课程班有 `teacher_id` 和 `teacher_name`。
- 效用边表：用 `student_course_utility_edges.csv` 表示每个学生对每个课程班的效用。
- 时间槽：使用离散时间段，例如 `Mon-1-2`、`Tue-3-4`。
- 轮内时间点：默认每轮5个时间点，最后一个时间点为截止时刻。
- 轮次开放：按配置控制，不写死在代码里。

## 3. 大模型学生代理

大模型在实验中扮演学生。每个学生代理输入：

- 当前轮次。
- 当前轮内时间点。
- 距离截止还有几个时间点。
- 当前可用预算。
- 已中选课程班。
- 本轮可投课程班列表。
- 每个课程班容量、剩余容量、当前待选人数、学分、类别、时间、是否必修、授课教师。
- 上一时间点投豆向量。
- 当前轮内待选人数历史。
- 学生对每个课程班的效用分解：兴趣、老师偏好、时间偏好、类别偏好、学分价值、必修完成收益。
- 未完成必修课程代码的惩罚。
- 学生风险类型：保守型、均衡型、激进型。
- 是否属于公式信息组。
- 当前可见信息：本轮实时待选人数和容量；不可见信息：其他人的投豆数和投豆分布。
- 历史信息：上一轮竞争情况、未中课程班、剩余课程机会。

大模型输出：

- 对每个课程班投多少豆。
- 决策理由。
- 对失败风险的判断。
- 对效用和豆子机会成本的权衡。
- 对当前待选人数和容量比的判断。
- 本次修改动作：保持、加豆、减豆、撤出课程班或新增投豆。
- 是否有意保留预算给后续轮次。

输出必须通过程序校验：

- 总投豆不得超过本轮可用预算。
- 不得对不可选课程班投豆。
- 豆子数必须为非负整数；小数、负数、字符串数字和空值都必须判为非法。
- 每个时间点的修改后投豆都必须满足预算约束。
- 截止后不允许修改投豆。
- 不得主动制造已知的课程代码重复、时间冲突和学分超限。
- 投豆理由不得把“多选课”当成目标，必须围绕效用、必修惩罚、风险和预算机会成本解释。

## 4. 对照策略

为了判断大模型策略是否有意义，需要设置非大模型基准：

- 均分策略：把预算均分给所有可行目标课程班。
- 效用比例策略：按 $u_{ic}$ 或正效用比例分配豆子。
- 必修惩罚优先策略：优先规避未完成必修课惩罚，再追求选修效用。
- 老师偏好策略：在同课程代码的不同课程班中优先投给老师效用最高的班。
- 保守策略：热门课少赌，优先容量宽松且效用稳定的课程班。
- 激进策略：集中豆子抢最高净效用课程班。
- 临近容量0豆策略：截止前若待选人数低于容量，把低豆投标改为0但保留待选状态。
- 最后一刻押入策略：截止前根据最新待选人数追加高效用课程班投豆。
- 公式信息策略：对信息组学生引入公式信号，但不把公式直接视为最优解。

大模型策略与这些基准在相同数据和随机种子下比较。

## 5. 实验流程

单轮简化实验：

1. 读取学生、课程班和效用边表数据。
2. 将单轮选课期离散为多个时间点。
3. 每个时间点更新学生可见的当前待选人数。
4. 学生可以保持、加豆、减豆、撤出课程班或新增投豆。
5. 截止时生成最终投豆方案。
6. 对每个课程班按最终投豆排序开奖。
7. 记录中选结果、消耗豆子、未中课程班和学生效用。
8. 统计不同策略的表现。

三轮现实实验：

1. 初始化每个学生预算为100。
2. 第1轮生成开放课程班和可投集合。
3. 将本轮选课期离散为多个时间点。
4. 为每个课程班生成或更新学生可见的当前待选人数。
5. 从效用边表读取本轮每个学生的可行课程班效用。
6. 学生代理在每个时间点根据容量、当前待选人数和历史投豆修改整数投豆。
7. 截止时锁定最终投豆，写入 `decisions.csv`。
8. 系统开奖，中选扣豆，未中退豆。
9. 更新已选课程班、剩余容量、剩余预算和可选集合。
10. 重复第2轮和第3轮。
11. 输出最终课表和指标。

公式信息冲击实验：

1. 按 `formula_informed_flag` 将约30%学生设为信息组。
2. 对信息组学生提供公式信号或由公式生成的竞争价格提示。
3. 若公式信号为小数，先按配置取整并截断到整数预算范围内，再允许进入投豆策略。
4. 比较信息组、非信息组和无公式基线的效用变化。
5. 检查公式是否抬高热门课录取边界并造成过度投豆。

## 6. 评价指标

个体层面：

- `total_utility`：最终课表总效用。
- `gross_course_utility`：不扣成本和惩罚的课程班效用总和。
- `unmet_required_penalty`：未完成必修课程代码产生的惩罚。
- `credits_taken`：最终获得学分。
- `budget_spent`：最终消耗豆子。
- `budget_left`：最终剩余豆子。
- `utility_per_bean`：单位豆子获得的净效用。
- `failed_high_utility_count`：高效用课程班失败数量。
- `regret_proxy`：用未选中高效用可行课程班估计的后悔值。
- `bid_revision_count`：轮内修改投豆次数。
- `last_minute_revision_count`：截止时间点修改投豆次数。

课程班层面：

- `demand_count`：申请人数。
- `observed_waitlist_count`：学生决策时可见的当前待选人数。
- `capacity`：课程班容量。
- `fill_rate`：容量填满率。
- `cutoff_bid`：录取边界投豆。
- `tie_break_count`：边界同分人数。
- `average_utility_of_applicants`：申请者对该课程班的平均效用。
- `waitlist_trajectory`：轮内待选人数变化轨迹。

系统层面：

- `average_total_utility`：学生平均最终效用。
- `required_completion_average`：平均必修完成率。
- `formula_group_utility_gap`：公式信息组与非信息组的平均效用差。
- `cutoff_bid_inflation`：公式信息引入后录取边界的变化。
- `overbidding_count`：投豆成本明显超过课程班效用收益的次数。
- `credit_cap_violation_count`：学分上限违规数，应为0。
- `time_conflict_violation_count`：时间冲突违规数，应为0。
- `code_duplicate_violation_count`：课程代码重复违规数，应为0。

## 7. 输出约定

每次实验建立独立目录：

```text
outputs/runs/<run_id>/
├─ decisions.csv
├─ bid_events.csv
├─ allocations.csv
├─ budgets.csv
├─ utilities.csv
├─ metrics.json
├─ config_snapshot.yaml
└─ llm_traces.jsonl
```

字段含义：

- `decisions.csv`：每个学生每轮对每个课程班投多少豆。
- `bid_events.csv`：每个学生在轮内每个时间点如何修改投豆。
- `allocations.csv`：每个课程班每轮的录取结果、边界投豆和同分抽签信息。
- `budgets.csv`：每个学生每轮开始预算、支付、退豆和轮后预算。
- `utilities.csv`：每个学生最终课表效用、惩罚、成本和净效用。
- `metrics.json`：本次实验聚合指标。
- `config_snapshot.yaml`：运行时配置快照，保证结果可复现。
- `llm_traces.jsonl`：大模型输入、输出、校验结果和修正记录。

`bid_events.csv` 记录过程，`decisions.csv` 只记录截止时最终投豆，开奖只使用 `decisions.csv`。`decisions.csv`、`allocations.csv` 和 `budgets.csv` 中所有豆子相关字段都必须是整数。若记录公式或大模型的连续建议，应另存为 `formula_signal` 或 trace 字段，不得覆盖最终整数投豆字段。

## 8. 第一批实验

建议第一批做六组：

- `simple_equal_split`：单轮，均分策略。
- `simple_utility_weighted`：单轮，效用比例策略。
- `simple_intra_round_dynamic`：单轮多时间点，测试撤豆和最后一刻押入。
- `realistic_baselines`：三轮，多个非大模型策略对照。
- `realistic_llm_agents`：三轮，大模型学生代理。
- `formula_30pct_raw`：30%学生直接接收公式信号。
- `formula_30pct_utility_filtered`：30%学生接收公式信号，但只有当效用足够高时才按公式加价。

等这些跑通后，再研究公式错误传播、不同学院之间信息不对称、公式信号被二次传播等复杂问题。

## 9. 验收测试场景

- 单轮多时间点：学生前几个时间点可以修改投豆，最终开奖只使用最后时间点。
- 整数约束：每次修改后的 `new_bid` 必须是非负整数，且学生总投豆不得超过当前预算。
- 低拥挤0豆：临近截止时若 `observed_waitlist_count < capacity`，保守策略可以把低豆投标改为0但保留待选状态。
- 高拥挤加豆：若 `observed_waitlist_count > capacity` 且课程班效用高，激进策略可以追加豆子。
- 不可见投豆：学生策略只能使用容量、待选人数和历史可见信息，不能读取其他学生投豆。
- 截止锁定：截止后不允许修改，开奖、扣豆和退豆均基于最终投豆。
